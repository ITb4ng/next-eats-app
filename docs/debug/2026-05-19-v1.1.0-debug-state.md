# v1.1.0 Debug State 정리

작성일: 2026-05-19

최근 정리: 2026-05-20

현재 작업 브랜치: `fix/v1.1.0-favicon-debug-state`


## 목적

이 문서는 메인 페이지의 로딩, 빈 데이터, 실패, 정상 상태를 개발 중에 일관되게 확인하기 위한 기준을 정리한다.
그리고 운영적인 관점에서 차후 장애 대응 메뉴얼의 기반 문서로 작성한다.


## 용어 분리

| 구분 | 목적 | 사용자에게 보임 | 예시 |
|---|---|---:|---|
| `debugState` | 사용자-facing 상태를 URL로 강제 미리보기 | 개발 환경에서만 보조 문구 표시 | `loading`, `empty`, `failure`, `success` |
| `debugReason` | 미리보기 또는 런타임 상태의 내부 원인을 보조 표시 | 개발 환경에서만 표시 | `KAKAO_MAP_SDK_TIMEOUT`, `store-network-error` |
| `[ops:...]` | 개발 환경 로그에서 검색 가능한 진단 코드 | 화면에는 표시하지 않음 | `[ops:DB_STORES_UNAVAILABLE]` |
| 실제 런타임 상태 | 브라우저, API, SDK, DB 결과로 자연 발생 | 사용자용 안내만 표시 | SDK script error, API 503, 빈 DB |

중요한 원칙은 "사용자 UI 상태는 적게, 내부 원인은 자세하게"다. 사용자는 `KAKAO_MAP_SDK_TIMEOUT`을 이해할 필요가 없고, 개발자와 운영자는 그 값을 알아야 빠르게 원인을 찾을 수 있다.

## 사용자 UI 상태 4개

| 상태 | 의미 | 사용자 화면의 역할 |
|---|---|---|
| `loading` | 지도, API, 마커 등 화면 구성 요소를 준비 중 | 멈춘 화면이 아니라 진행 중임을 알려준다. |
| `empty` | 요청은 성공했지만 표시할 데이터가 없음 | 사용자가 다음 행동을 알 수 있게 한다. 예: 맛집 등록 안내 |
| `failure` | SDK, 네트워크, API, DB 등 핵심 흐름 실패 | 문제를 인식시키고 재시도 또는 대체 경로를 제공한다. |
| `success` | 정상 화면 표시 가능 | 실제 지도, 마커, 가게 정보를 표시한다. |

`empty`와 `failure`는 다르다.

- `empty`: 시스템은 정상이다. 보여줄 데이터가 아직 없다.
- `failure`: 시스템 또는 외부 의존성이 정상 응답하지 않았다.

이 차이가 UI에서 드러나야 사용자가 "내가 등록해야 하는 상황"인지, "잠시 후 다시 시도해야 하는 상황"인지 구분할 수 있다.

## 메인 페이지 기준 상태

메인 페이지는 지도 중심 서비스이므로 Kakao Map을 불러오지 못하면 첫 화면의 핵심 역할이 크게 약해진다. 따라서 지도 실패는 단순한 작은 toast가 아니라 메인 영역에서 명확히 처리해야 한다.

| 상황 | 현재 의미 | fallback 방향 |
|---|---|---|
| 지도 SDK/script 실패 | 메인 맵 사용 불가 | 지도 영역을 대체하는 명확한 실패 UI, 재시도, 가게 목록 대체 경로 |
| 지도는 열림, `/api/stores` 실패 | 지도 shell은 가능하지만 마커 데이터 불가 | 지도는 유지하되 데이터 문제를 눈에 띄게 안내, 재시도 제공 |
| 지도는 열림, store 데이터 0개 | 장애가 아니라 등록 데이터 없음 | empty UI와 맛집 등록 CTA 제공 |
| 지도는 열림, Supabase/DB paused 또는 지연 | 데이터 의존성 장애 | 단순 네트워크 실패보다 "데이터 응답 지연"으로 설명, 재시도 제공 |
| 지도는 열림, 좌표 없는 데이터만 있음 | 가게는 있으나 마커 표시 불가 | 위치 정보 부족을 안내하고 목록 확인 또는 수정 흐름 제공 |

이 문서의 다음 단계 작업은 위 기준을 바탕으로 B, C fallback UI를 실제 화면에 반영하는 것이다.

## 확인 방법 1: URL Debug Preview

URL debug preview는 UI 문구와 레이아웃을 빠르게 확인하기 위한 방법이다. 실제 네트워크 요청을 차단하거나 DB 장애를 만들지는 않는다.

개발 환경에서만 동작한다.

```text
http://localhost:3000/?debugState=loading
http://localhost:3000/?debugState=empty
http://localhost:3000/?debugState=failure
http://localhost:3000/?debugState=success
```

현재 구현 기준:

- `loading`, `empty`, `failure`는 홈의 실제 지도/API/마커 렌더링을 멈추고 통합 preview UI를 보여준다.
- `success`는 preview 화면을 만들지 않고 정상 흐름을 그대로 진행한다.
- production build에서는 `debugState` query를 넣어도 강제 preview가 동작하지 않아야 한다.

구체 원인을 같이 표시하고 싶으면 `debugReason`을 붙일 수 있다.

```text
http://localhost:3000/?debugState=failure&debugReason=KAKAO_MAP_SDK_TIMEOUT
http://localhost:3000/?debugState=failure&debugReason=store-network-error
http://localhost:3000/?debugState=empty&debugReason=marker-empty
```

주의할 점:

- 위 URL은 "통합 상태 preview"다.
- 실제 Kakao SDK 실패, 실제 `/api/stores` 실패, 실제 Supabase paused 상태를 재현하는 방법은 아니다.
- 실제 실패 UI 배치까지 확인하려면 아래 DevTools request blocking 또는 런타임 테스트를 사용한다.

## 확인 방법 2: 실제 장애 재현

실제 장애 재현은 브라우저와 앱 코드가 실패 경로를 실제로 타는지 확인하는 방법이다.

Chrome / Edge 기준:

1. `http://localhost:3000/` 접속
2. DevTools 열기
3. `Network` 탭 열기
4. `Network request blocking` 열기
5. `Enable network request blocking` 체크
6. 필요한 차단 패턴 추가
7. 페이지 새로고침

자주 쓰는 차단 패턴:

```text
*://dapi.kakao.com/*
*://map*.daumcdn.net/*
*://*.daumcdn.net/*
*://localhost:3000/api/stores*
```

차단 패턴별 의미:

| 차단 패턴 | 확인하려는 상태 | 기대되는 UI |
|---|---|---|
| `*://dapi.kakao.com/*` | Kakao Map SDK script 실패 | 지도 영역의 blocking fallback |
| `*://map*.daumcdn.net/*` | 지도 tile 지연/실패 | 지도 좌측 상단의 작은 지연 안내 |
| `*://localhost:3000/api/stores*` | Store API 네트워크 실패 | 가게 데이터 실패 안내와 재시도 |
| `dapi.kakao.com` + `/api/stores` 동시 차단 | 지도 실패와 데이터 실패 동시 발생 | 지도 fallback과 store 실패 안내가 역할별로 분리되어야 함 |

실제 장애 재현에서는 URL debug preview를 함께 쓰지 않는 편이 좋다. `debugState=loading|empty|failure`가 실제 렌더링을 멈추기 때문에, DevTools 차단으로 발생하는 실제 흐름을 가릴 수 있다.

## 확인 방법 3: API와 DB 상태 확인

Store API 직접 확인:

```text
http://localhost:3000/api/stores
http://localhost:3000/api/stores?q=__debug_no_result__
```

`/api/stores?q=__debug_no_result__`는 API가 정상 응답하지만 결과가 0개인 empty 흐름을 확인하는 데 쓴다. 홈 화면의 실제 empty UI와 완전히 같지는 않을 수 있으므로, 최종 UI 확인은 메인 페이지에서 해야 한다.

DB 장애 경로 확인:

```bash
npm run test:perf:v1.1.0 -- --runtime-db
```

Kakao SDK script error 경로 확인:

```bash
npm run test:perf:v1.1.0 -- --browser-sdk
```

`--browser-sdk`는 Playwright가 설치된 환경에서만 실행한다.

## Map Debug State

| Debug state | 발생 조건 | 사용자에게 줄 메시지 방향 | 확인 방법 |
|---|---|---|---|
| `KAKAO_MAP_CREATING` | SDK ready 이후 지도 인스턴스 생성 중 | 지도 화면을 준비 중임을 짧게 안내 | `/` 진입 직후 또는 `?debugReason=KAKAO_MAP_CREATING` |
| `KAKAO_MAP_SDK_MISSING_CONFIG` | Kakao Map public key 없음 | 설정 문제로 지도를 표시하지 못함 | Kakao public env 없이 실행 |
| `KAKAO_MAP_SDK_TIMEOUT` | SDK 응답 10초 초과 | 지도 응답 지연, 잠시 후 재시도 안내 | 느린 네트워크, offline, 테스트 |
| `KAKAO_MAP_SDK_SCRIPT_ERROR` | SDK script 로딩 실패 | 지도 서비스를 불러오지 못함 | `*://dapi.kakao.com/*` 차단 |
| `KAKAO_MAP_SDK_INIT_ERROR` | script는 로드됐지만 `window.kakao.maps.load` 초기화 실패 | 지도 초기화 실패 | 비정상 SDK 응답 상황 |
| `KAKAO_MAP_CREATE_ERROR` | `new maps.Map(...)` 생성 중 예외 | 지도 화면 생성 실패 | container 또는 SDK namespace 이상 |
| `KAKAO_MAP_TILE_TIMEOUT` | 지도는 열렸지만 tile 응답 8초 초과 | 기본 지도는 열렸고 일부 타일이 늦을 수 있음 | tile 도메인 차단 또는 느린 네트워크 |

지도 SDK/script 실패는 메인 페이지 핵심 기능 실패다. B 작업에서는 이 상태를 "작은 에러 카드"보다 더 명확한 대체 경험으로 다루는 것이 맞다.

## Store Debug State

| Debug state | 발생 조건 | 사용자에게 줄 메시지 방향 | 확인 방법 |
|---|---|---|---|
| `store-loading` | `/api/stores` 요청 중 | 가게 정보를 불러오는 중 | `/` 진입 직후 |
| `store-empty` | API 성공, 데이터 0개 | 등록된 가게가 없고 등록이 필요함 | 빈 DB 또는 empty API 응답 |
| `store-database-delay` | DB timeout, gateway, Supabase pause, Prisma 연결류 오류 | 데이터 응답 지연, 잠시 후 재시도 | runtime DB 테스트 또는 DB 연결 장애 |
| `store-network-error` | 브라우저에서 `/api/stores` 응답을 받지 못함 | 네트워크 연결 확인, 재시도 | `/api/stores` 차단 |
| `store-api-error` | DB 지연류가 아닌 API 4xx/5xx 오류 | 가게 정보를 불러오지 못함 | API 일반 예외 |

Store 실패는 지도가 열렸는지 여부에 따라 배치가 달라질 수 있다.

- 지도가 정상이라면 지도 위 또는 주변에 데이터 문제를 명확히 표시한다.
- 지도도 실패했다면 지도 fallback 안에서 "지도 문제"와 "가게 데이터 문제"가 섞이지 않게 분리한다.
- `store-empty`는 장애가 아니므로 실패 색상보다 등록 유도와 다음 행동이 중심이어야 한다.

## Marker Debug State

| Debug state | 발생 조건 | 사용자에게 줄 메시지 방향 | 확인 방법 |
|---|---|---|---|
| `marker-empty` | 지도는 열렸지만 표시 가능한 좌표가 없음 | 위치 정보가 있는 가게가 없음을 안내 | 가게 데이터는 있으나 `lat/lng` 없음 |
| `marker-partial-error` | 일부 마커 생성 실패 | 일부 위치만 표시되지 않았고 목록은 확인 가능 | 일부 좌표/마커 이미지 문제 |
| `marker-error` | 모든 마커 생성 실패 | 가게 위치 표시 실패, 목록 대체 확인 | SDK map namespace 이상 또는 전체 마커 실패 |

Marker 문제는 지도 SDK 실패와 다르다. 지도 자체는 열렸으므로 메인 기능 전체 실패라기보다 "가게 위치 시각화 실패"에 가깝다.

## Console / Server Log 확인

개발 환경의 브라우저 콘솔 또는 서버 로그에서 아래 형식으로 검색한다. production에서는 브라우저 콘솔에 `[ops:*]` 진단 코드가 노출되면 안 된다.

```text
[ops:KAKAO_MAP_SDK_MISSING_CONFIG]
[ops:KAKAO_MAP_SDK_TIMEOUT]
[ops:KAKAO_MAP_SDK_INIT_ERROR]
[ops:KAKAO_MAP_SDK_SCRIPT_ERROR]
[ops:DB_STORES_UNAVAILABLE]
[ops:STORE_API_ERROR]
```

로그의 역할:

- 사용자 화면 문구를 길고 기술적으로 만들지 않는다.
- 개발 환경에서는 `[ops:...]` 코드로 개발자가 원인을 추적한다.
- production 브라우저에서는 네트워크 차단, SDK 실패, API 실패가 발생해도 `[ops:...]` 코드를 출력하지 않는다.
- 실제 env 값, API key, DB URL은 로그나 문서에 기록하지 않는다.

## 개발/프로덕션 분리 기준

개발 환경:

- `debugState` query 허용
- `debugReason` query 허용
- 화면에 `debug:` 보조 문구 표시 가능
- `[ops:*]` 진단 로그 출력 가능
- React Query Devtools 표시 가능

프로덕션:

- `debugState` query가 UI를 강제로 바꾸면 안 됨
- `debugReason` query가 화면에 노출되면 안 됨
- `debug:` 보조 문구가 보이면 수정 대상
- 브라우저 콘솔에 `[ops:*]` 진단 로그가 보이면 수정 대상
- 사용자는 복구 가능한 안내와 다음 행동만 봐야 함

이 분리가 필요한 이유는 production에서 URL query 하나로 실패 화면을 만들 수 있으면 서비스 신뢰도와 보안에 모두 좋지 않기 때문이다. 내부 장애 코드가 사용자에게 그대로 노출되는 것도 피해야 한다.

## 추천 수동 점검 순서

1. `npm run dev` 실행
2. `http://localhost:3000/?debugState=loading`으로 통합 loading preview 확인
3. `http://localhost:3000/?debugState=empty`로 통합 empty preview 확인
4. `http://localhost:3000/?debugState=failure`로 통합 failure preview 확인
5. `http://localhost:3000/?debugState=success`로 정상 흐름이 유지되는지 확인
6. URL query를 제거하고 `http://localhost:3000/`으로 접속
7. DevTools에서 `*://dapi.kakao.com/*` 차단 후 지도 SDK 실패 UI 확인
8. DevTools에서 `*://localhost:3000/api/stores*` 차단 후 Store API 실패 UI 확인
9. 두 차단을 동시에 켜고 지도 실패와 Store 실패가 섞이지 않는지 확인
10. `cmd /c npm run build` 실행
11. `npm run start` 또는 `cmd /c npm run start` 실행
12. production build에서 `?debugState=failure&debugReason=store-network-error`를 넣어도 debug UI가 강제 주입되지 않는지 확인
13. production build에서 `/api/stores` 또는 Kakao SDK 요청을 차단해도 브라우저 콘솔에 `[ops:*]` 로그가 남지 않는지 확인

## B/C 작업 전 결정해야 할 UX 기준

B. 맵을 못 불러왔을 경우:

- 메인 페이지 핵심 기능 실패로 본다.
- 지도 영역의 작은 fallback 카드만으로 충분한지 재검토한다.
- 최소한 재시도, 문제 설명, 가게 목록 대체 탐색 경로가 필요하다.

C. 맵은 열렸지만 Store 데이터가 없거나 실패한 경우:

- `store-empty`: 장애가 아니다. 맛집 등록 CTA가 중심이어야 한다.
- `store-database-delay`: Supabase paused/free tier 한계 가능성을 사용자가 이해할 수 있는 말로 안내한다.
- `store-network-error`: 사용자 네트워크 또는 요청 차단 가능성을 안내하고 재시도를 제공한다.
- `marker-empty`: 데이터는 있으나 좌표가 없어 지도에 표시하지 못하는 상태로 분리한다.

| 상태 조합 | 기존 화면/문제 | 변경 후 기준 | tone |
|---|---|---|---|
| 맵 실패 + DB/API 성공 | 지도 아래 실제 가게 10개 미리보기 fallback 표시. 메인 페이지 역할이 흐려짐 | 10개 미리보기 제거. 지도 중앙 카드에서 지도만 실패했고 서비스는 계속 이용 가능하다고 안내. `/stores` CTA 제공 | `warning` |
| 맵 실패 + DB/API 실패 | 지도 실패와 데이터 실패가 서로 분리되어 보이나 전체 탐색 제한 맥락이 약함 | 지도 중앙 카드에 탐색 기능 제한 안내 추가. DB/API 실패는 별도 toast 유지 | `critical` |
| 맵 성공 + DB/API 실패 | 지도는 정상인데 Store 실패 toast가 빨강 alert로 강하게 보임 | 지도는 유지. Store 실패 toast를 주황 warning으로 낮춰 표시 | `warning` |

fallback tone 기준:

- `warning`: 지도 또는 DB/API 중 하나만 실패한 상태
- `critical`: 지도와 DB/API가 동시에 실패한 상태
- 현재 class 기준은 `src/lib/fallbackTone.ts`에 모아두었고, 추후 컬러 시스템 토큰으로 승격할 수 있다.

D 작업은 B/C UI와 별개의 production 노출 분리 작업이다.

- development: `debugState`, `debugReason`, `debug:`, `[ops:*]` 진단 로그 허용
- production: URL debug preview, debug 보조 문구, `[ops:*]` 브라우저 콘솔 노출 금지

## 주의 사항

- URL debug preview와 실제 장애 재현을 섞어서 해석하지 않는다.
- 아이콘, manifest, favicon 확인은 이 문서의 debug state 범위가 아니다.
- 실제 env 값은 로그나 문서에 기록하지 않는다.
- Kakao Map SDK와 지도 타일은 외부 리소스라 앱 코드만으로 전부 통제할 수 없다.
- retry, submit, back 버튼 스타일은 후속 공통 Button 스타일 정리 작업에서 통합한다.
