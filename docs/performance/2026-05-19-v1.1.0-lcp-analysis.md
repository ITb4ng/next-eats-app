# v1.1.0 LCP 원인 후보 분석

작성일: 2026-05-19

작업 기준 브랜치: `perf/v1.1.0-lcp-analysis`

## 분석 목적

`v1.0.0` 대비 초기 Production 성능 비교 문서화 이후, LCP가 약 `2.1s` 수준으로 유지되던 원인을 코드 구조 기준으로 분석한다.

이번 문서는 최적화 작업을 바로 수행하기 위한 문서가 아니라, `v1.1.0` 로딩 안정화에서 어떤 항목을 먼저 다룰지 정리하기 위한 분석 기록이다. 기능 코드, 성능 최적화 코드, 리팩토링 코드, 스타일 코드는 수정하지 않았다.

## 현재 기준 성능 수치 요약

| 항목 | `v1.0.0` 배포 버전 | 초기 Production 기준 | `v1.1.0` Production 최종 |
|---|---:|---:|---:|
| Desktop Performance 평균 | 78.66 | 88 | 96 |
| Mobile Performance 1회 측정 | 약 72 | 75 | 추가 반복 측정 필요 |
| SEO | 54 | 91 | 100 |
| LCP | 약 2.1s | 약 2.1s | 약 1.4s |

초기 Production 문서 기준으로는 Desktop Performance 평균과 SEO는 개선되었지만, LCP는 `v1.0.0`과 현재 Production 모두 약 `2.1s` 수준으로 확인되었다. 이후 PR #9 `perf: v1.1.0 로딩 리소스 최적화 및 SEO 보완` 반영 뒤 Production 최종 측정에서는 Performance `96`, SEO `100`, LCP 약 `1.4s`를 확인했다.

이 결과는 측정 시점의 Lighthouse 기준 개선 확인이며, 항상 같은 수치를 보장한다는 의미는 아니다. `v1.1.0` 기준 로딩 안정화 목표는 달성했고, 추가 최적화는 자동화 측정 기반으로 회귀를 관리하는 방향으로 본다.

## Production 기준과의 차이

- 기존 Production 문서 기준 LCP는 약 `2.1s`였다.
- 이번 local build/start 측정에서는 LCP가 `1.4s`로 확인되었다.
- 다만 local 측정값은 Production 배포 환경과 다르므로, Production 기준값을 바로 대체하지 않는다.
- local 측정 결과는 코드가 production build 상태에서 더 안정적으로 동작하는지 확인하는 참고 지표로 사용한다.
- Production URL에서 같은 조건으로 재측정해야 실제 개선 여부를 확정할 수 있다.

## LCP element 확인

- LCP element는 `div#map`으로 확인되었다.
- selector는 `div.layout > main.relative > section.relative > div#map`이다.
- LCP element class는 `h-[calc(100dvh-var(--navbar-height))] w-full bg-slate-100`이다.
- 현재 메인페이지에서 지도 컨테이너가 가장 큰 시각 요소로 인식된다.
- 따라서 LCP 개선은 단순 이미지 최적화보다 지도 컨테이너의 초기 표시 방식, Kakao Map SDK 로딩 타이밍, fallback UI 표시 방식과 관련이 있다.

## LCP breakdown 해석

| Subpart | Duration |
|---|---:|
| Time to First Byte | 약 6.2ms |
| Resource Load Delay | 약 213.2ms |
| Resource Load Duration | 약 8.6ms |
| Element Render Delay | 약 46.6ms |

- TTFB는 매우 낮아 서버 응답 자체는 병목으로 보기 어렵다.
- Element Render Delay도 크지 않다.
- 상대적으로 Resource Load Delay가 가장 크므로 Kakao Map SDK, 지도 타일, 외부 리소스 요청 타이밍을 계속 관찰해야 한다.
- 다만 전체 LCP가 `1.4s`로 양호하므로 대규모 구조 변경보다 측정 반복과 안전한 개선이 우선이다.

## Best Practices 해석

- Best Practices 점수는 `77`로 낮게 보이지만, local HTTP 측정의 영향이 크다.
- `localhost:3000`이 HTTP로 실행되면서 Kakao Map 관련 HTTP 요청이 Lighthouse에 감점 요소로 잡혔다.
- Production은 HTTPS 배포 환경이므로 이 결과를 그대로 Production 문제로 단정하지 않는다.
- Production URL에서 Best Practices를 별도로 재측정해야 한다.

## unused JavaScript 해석

- unused JavaScript 항목에는 Chrome extension script가 포함되어 있다.
- 따라서 다음 측정은 확장 프로그램이 없는 시크릿 모드 또는 별도 Chrome 프로필에서 수행해야 한다.
- Kakao Map JS와 일부 Next chunk도 unused 후보로 잡혔으나, 현재 LCP와 TBT가 양호하므로 즉시 대규모 dynamic import 리팩토링으로 이어가지는 않는다.
- 이후 Production 측정에서 동일하게 반복될 경우 지도 컴포넌트 분리 또는 SDK 로딩 타이밍 개선을 검토한다.

## Preview SEO 54 원인

Lighthouse SEO 감점 항목은 다음 두 가지였다.

- Page is blocked from indexing
- Document does not have a meta description

코드와 설정 기준으로 확인한 결과는 다음과 같다.

| 확인 항목 | 결과 |
|---|---|
| `src/pages/_document.tsx` | `meta name="robots"` 또는 `noindex` 설정 없음 |
| `src/pages/_app.tsx` | favicon, preconnect, title, viewport 설정만 있으며 robots 관련 meta 없음 |
| `src/pages/index.tsx` | 홈 페이지 description이 없었고, 이번 작업에서 `meta name="description"`을 추가했다. |
| `next.config.ts` | 정적 리소스 cache header와 favicon redirect만 있으며 `X-Robots-Tag` 없음 |
| `vercel.json` | build env만 있으며 headers 또는 robots 관련 설정 없음 |
| `public` | `robots.txt` 파일 없음 |
| repository 검색 | `noindex`, `X-Robots-Tag`, `Disallow`, `meta robots` 설정 없음 |

현재 repository 기준으로는 Preview URL에 indexing 차단을 명시하는 코드나 설정을 확인하지 못했다. 따라서 Preview의 `Page is blocked from indexing`은 Vercel Preview 배포 특성 또는 배포 플랫폼/프로젝트 설정에서 적용된 가능성이 높다.

다만 Production URL에 같은 `noindex` 또는 `X-Robots-Tag`가 적용되는지는 이 문서만으로 확정하지 않는다. Production URL에서 응답 header와 Lighthouse를 별도로 재측정해야 한다.

## Preview와 Production SEO 측정 구분

- Preview URL은 검색 노출 대상이 아니므로 noindex가 적용될 수 있다.
- Preview의 SEO `54`를 Production SEO 문제로 단정하지 않는다.
- Production URL에서는 indexing 차단 여부를 별도로 확인해야 한다.
- Production에서도 `meta name="robots"` 또는 `X-Robots-Tag: noindex`가 확인되면 수정 후보로 분리한다.
- meta description 누락은 Preview 특성과 무관한 실제 보완 대상이므로 홈 페이지에 description을 추가했다.

## Production SEO 재측정 필요

Production URL에서는 다음 항목을 다시 확인해야 한다.

- Lighthouse SEO 점수 3회 반복 측정
- `Document does not have a meta description` 항목 해소 여부
- `Page is blocked from indexing` 항목 발생 여부
- 응답 header에 `X-Robots-Tag`가 있는지 확인
- HTML에 `meta name="robots"`가 있는지 확인
- `/robots.txt`가 존재하거나 `Disallow`를 반환하는지 확인

## 추가 최적화 판단

Production LCP가 약 `1.4s` 수준으로 확인되었기 때문에, `v1.1.0` 단계에서 추가적인 대규모 LCP 최적화는 우선순위가 낮다. 이후에는 Playwright MCP 또는 반복 Lighthouse 측정을 통해 성능 회귀를 감지하고, 모바일 성능이나 지도 fallback 상태에서 문제가 반복될 경우 작은 범위의 최적화를 진행한다.

Kakao Map 관련 외부 리소스는 앱 코드만으로 전부 통제하기 어렵다. debug state, fallback state, loading state의 production 안전성은 별도 점검 대상으로 유지하되, SDK 로딩 방식이나 지도 중심 구조의 전면 변경은 `v1.1.0`의 즉시 과제로 보지 않는다.

## LCP가 공통 개선 대상인 이유

LCP는 초기 화면에서 가장 큰 콘텐츠가 사용자에게 보이는 시점을 측정한다. 현재 홈 화면은 전체 화면 지도 영역을 먼저 배치하고, 지도 SDK, 지도 타일, 마커, 가게 데이터, 폰트가 함께 초기 화면 경험에 영향을 주는 구조다.

`v1.0.0` 이후 fallback UI와 SEO는 개선되었지만, 지도 중심 초기 렌더링 구조 자체는 크게 바뀌지 않았다. 초기 Production에서 LCP가 약 `2.1s` 수준으로 유지된 원인은 특정 단일 코드보다 다음 요소가 결합된 결과일 가능성이 크다.

- 전체 화면 Kakao Map 영역이 홈 첫 화면의 핵심 콘텐츠로 렌더링된다.
- Kakao Map SDK와 지도 타일은 외부 네트워크에 의존한다.
- 홈 화면은 서버 렌더링 데이터 주입 없이 클라이언트에서 `/api/stores`를 요청한다.
- 마커 이미지는 카테고리별 PNG 파일을 사용하며 일부 파일 크기가 크다.
- 상단 네비게이션과 메뉴에 로컬 TTF 폰트가 적용되어 초기 텍스트 렌더링에 영향을 줄 수 있다.

## Kakao Map 관련 병목 후보

### SDK 로드 방식

`useKakaoMapSdk`는 Next.js `Script` 컴포넌트가 아니라 `document.createElement("script")`로 SDK 스크립트를 직접 생성한다.

- SDK URL에 `autoload=false`가 포함되어 있다. 근거: `src/hooks/useKakaoMapSdk.ts:23-26`
- script는 `async = true`로 설정된다. 근거: `src/hooks/useKakaoMapSdk.ts:131`
- script는 `document.head.appendChild(scriptElement)`로 삽입된다. 근거: `src/hooks/useKakaoMapSdk.ts:134`
- SDK load 이후 `window.kakao.maps.load(finishReady)`를 호출한다. 근거: `src/hooks/useKakaoMapSdk.ts:89`
- SDK timeout은 `10000ms`로 설정되어 있다. 근거: `src/hooks/useKakaoMapSdk.ts:12`, `src/hooks/useKakaoMapSdk.ts:74-82`

`autoload=false` 사용 자체는 긍정적이다. 다만 홈 첫 화면에서 지도 컴포넌트가 즉시 렌더링되므로, SDK 로드와 지도 초기화가 초기 화면의 주요 경로에 포함된다.

### 지도 초기화와 타일 로딩

`Map` 컴포넌트는 SDK가 ready가 되면 즉시 `new maps.Map(...)`으로 지도 인스턴스를 생성한다.

- 지도 생성: `src/components/Map.tsx:90-93`
- `tilesloaded` 이벤트에서 `tileStatus`와 `mapStatus`를 ready로 변경한다. 근거: `src/components/Map.tsx:100-106`
- `tilesloaded` 이벤트 리스너를 등록한다. 근거: `src/components/Map.tsx:119`
- 타일 timeout은 `8000ms`다. 근거: `src/components/Map.tsx:23`, `src/components/Map.tsx:123-126`

지도 타일은 외부 리소스이므로 앱 코드만으로 전부 통제하기 어렵다. 다만 현재 UI에서는 전체 화면 지도 영역이 첫 화면의 중심이므로, SDK와 타일 응답 시간이 LCP 근처 지표에 영향을 줄 가능성이 높다.

### fallback 구조

지도 SDK loading 또는 map creating 상태에서는 지도 위에 큰 상태 패널을 보여준다.

- SDK loading/map creating 조건: `src/components/Map.tsx:162-165`
- loading 패널 렌더링: `src/components/Map.tsx:187-194`
- 지도 실패 시 error 패널 표시: `src/components/Map.tsx:198-224`
- 타일 timeout 시 안내 표시: `src/components/Map.tsx:228-232`

fallback store list는 지도 실패가 확정되고, store fetch가 성공했을 때만 나타난다. 근거: `src/pages/index.tsx:252`, `src/pages/index.tsx:272`

따라서 "지도보다 store list 또는 fallback store가 먼저 사용자에게 보이는 구조"는 아니다. 현재 구조는 지도 실패를 감지한 뒤에야 list fallback이 노출된다. LCP 관점에서는 지도 로딩 패널이 먼저 큰 영역으로 잡힐 가능성이 있다.

| 운영용 장애 로그 코드 | DB 장애와 Kakao Map SDK 장애에 `[ops:...]` 형식의 고정 로그 코드를 추가했다. 사용자에게는 일반 메시지를 보여주고, 운영자는 서버/브라우저 콘솔에서 원인을 바로 필터링할 수 있다. | 완료 |

### v1.1.0 자동화 테스트 파일

자동화 테스트 진입점은 `scripts/performance-v1.1.0-runtime-checks.mjs`다.

기본 실행:

```bash
npm run test:perf:v1.1.0
```

기본 실행은 다음 운영 로그 계약을 확인한다.

- `/api/stores` DB 장애 경로에 `[ops:DB_STORES_UNAVAILABLE]` 로그 코드가 존재하는지 확인한다.
- `/api/stores` 일반 예외 경로에 `[ops:STORE_API_ERROR]` 로그 코드가 존재하는지 확인한다.
- Kakao Map SDK 설정 누락 경로에 `[ops:KAKAO_MAP_SDK_MISSING_CONFIG]` 로그 코드가 존재하는지 확인한다.
- Kakao Map SDK script error, timeout, init error 경로에 각각 `[ops:KAKAO_MAP_SDK_SCRIPT_ERROR]`, `[ops:KAKAO_MAP_SDK_TIMEOUT]`, `[ops:KAKAO_MAP_SDK_INIT_ERROR]` 로그 코드가 존재하는지 확인한다.

DB 연결 장애 런타임 확인:

```bash
npm run test:perf:v1.1.0 -- --runtime-db
```

이 모드는 의도적으로 잘못된 `DATABASE_URL`로 개발 서버를 띄운 뒤 `/api/stores`가 `503`을 반환하고 서버 로그에 `[ops:DB_STORES_UNAVAILABLE]`가 찍히는지 확인한다.

Kakao SDK 브라우저 런타임 확인:

```bash
npm run test:perf:v1.1.0 -- --browser-sdk
```

이 모드는 Playwright가 설치된 환경에서 Kakao SDK 요청을 차단하고, 브라우저 콘솔에 `[ops:KAKAO_MAP_SDK_SCRIPT_ERROR]`가 찍히는지 확인한다. 현재 프로젝트 devDependency에는 Playwright를 추가하지 않았으므로, CI 또는 로컬 측정 환경에 Playwright가 있는 경우에만 실행한다.

### 운영 핫픽스 관점

`[ops:...]` 로그 코드는 운영자가 Cloud/Vercel 로그, 브라우저 원격 로그, 장애 리포트에서 바로 검색할 수 있는 최소 진단 단위다. 현재 추가한 코드는 다음과 같다.

| 코드 | 의미 | 우선 확인 |
|---|---|---|
| `DB_STORES_UNAVAILABLE` | `/api/stores`에서 DB 연결, timeout, gateway, Supabase pause류 장애가 발생했다. | Supabase 상태, `DATABASE_URL`, connection pool, cold start |
| `STORE_API_ERROR` | `/api/stores`에서 DB 지연류로 분류되지 않은 예외가 발생했다. | 요청 method, API 코드 변경, Prisma 예외 |
| `KAKAO_MAP_SDK_MISSING_CONFIG` | 클라이언트에 Kakao Map app key가 주입되지 않았다. | `NEXT_PUBLIC_KAKAO_MAP_API_KEY`, 배포 환경 변수 |
| `KAKAO_MAP_SDK_SCRIPT_ERROR` | Kakao SDK script 로딩 자체가 실패했다. | 외부 네트워크, CSP, SDK URL, Kakao 장애 |
| `KAKAO_MAP_SDK_TIMEOUT` | Kakao SDK 응답이 timeout을 넘겼다. | 모바일 네트워크, Kakao 응답 지연 |
| `KAKAO_MAP_SDK_INIT_ERROR` | SDK script는 왔지만 `window.kakao.maps.load` 또는 namespace 초기화가 실패했다. | SDK 응답 내용, app key 권한, 도메인 등록 |

## 결론

초기 Production 기준 LCP 약 `2.1s`가 유지된 가장 강한 후보는 전체 화면 Kakao Map 중심 초기 렌더링 구조와 외부 지도 SDK/타일 의존성이었다. 여기에 고용량 favicon, 일부 고용량 marker 이미지, 로컬 TTF 폰트, 초기 JS chunk 후보가 함께 모바일 성능과 체감 로딩에 영향을 줄 수 있었다.

PR #9 이후 Production 측정에서는 Performance `96`, SEO `100`, LCP 약 `1.4s`를 확인했다. 따라서 `v1.1.0` 기준 로딩 안정화 목표는 달성한 것으로 정리하되, 이 결과를 최종적인 성능 보증으로 보지는 않는다.

앞으로는 지도와 목록의 구조적 재배치보다 반복 Lighthouse 측정, 모바일 기준선 정리, Playwright MCP 기반 시나리오 측정, debug/fallback/loading 상태의 production 안전성 점검을 우선한다. 큰 구조 변경은 `v2.0.0`에서 별도 설계 과제로 다루는 편이 안전하다.
