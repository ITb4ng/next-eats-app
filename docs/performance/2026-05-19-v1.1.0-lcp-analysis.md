# v1.1.0 LCP 원인 후보 분석

작성일: 2026-05-19

작업 기준 브랜치: `perf/v1.1.0-lcp-analysis`

## 분석 목적

`v1.0.0` 대비 현재 Production 성능 비교 문서화 이후, 현재 Production에서 LCP가 약 `2.1s` 수준으로 유지되는 원인을 코드 구조 기준으로 분석한다.

이번 문서는 최적화 작업을 바로 수행하기 위한 문서가 아니라, `v1.1.0` 로딩 안정화에서 어떤 항목을 먼저 다룰지 정리하기 위한 분석 기록이다. 기능 코드, 성능 최적화 코드, 리팩토링 코드, 스타일 코드는 수정하지 않았다.

## 현재 기준 성능 수치 요약

| 항목 | `v1.0.0` 배포 버전 | 현재 Production | 변화 |
|---|---:|---:|---:|
| Desktop Performance 평균 | 78.66 | 88 | +9.34 |
| Mobile Performance 1회 측정 | 약 72 | 75 | +3 |
| SEO | 54 | 91 | +37 |
| LCP | 약 2.1s | 약 2.1s | 큰 변화 없음 |

Desktop Performance 평균과 SEO는 개선되었지만, LCP는 `v1.0.0`과 현재 Production 모두 약 `2.1s` 수준으로 확인되었다. 따라서 LCP는 현재 Production의 공통 개선 대상으로 본다.

## LCP가 공통 개선 대상인 이유

LCP는 초기 화면에서 가장 큰 콘텐츠가 사용자에게 보이는 시점을 측정한다. 현재 홈 화면은 전체 화면 지도 영역을 먼저 배치하고, 지도 SDK, 지도 타일, 마커, 가게 데이터, 폰트가 함께 초기 화면 경험에 영향을 주는 구조다.

`v1.0.0` 이후 fallback UI와 SEO는 개선되었지만, 지도 중심 초기 렌더링 구조 자체는 크게 바뀌지 않았다. 따라서 LCP가 약 `2.1s` 수준으로 유지되는 원인은 특정 단일 코드보다 다음 요소가 결합된 결과일 가능성이 크다.

- 전체 화면 Kakao Map 영역이 홈 첫 화면의 핵심 콘텐츠로 렌더링된다.
- Kakao Map SDK와 지도 타일은 외부 네트워크에 의존한다.
- 홈 화면은 서버 렌더링 데이터 주입 없이 클라이언트에서 `/api/stores`를 요청한다.
- 마커 이미지는 카테고리별 PNG 파일을 사용하며 일부 파일 크기가 크다.
- 상단 네비게이션과 메뉴에 로컬 TTF 폰트가 적용되어 초기 텍스트 렌더링에 영향을 줄 수 있다.

## Kakao Map 관련 병목 후보

### SDK 로드 방식

`useKakaoMapSdk`는 Next.js `Script` 컴포넌트가 아니라 `document.createElement("script")`로 SDK 스크립트를 직접 생성한다.

- SDK URL에 `autoload=false`가 포함되어 있다. 근거: `src/hooks/useKakaoMapSdk.ts:23-26`
- script는 `async = true`로 설정된다. 근거: `src/hooks/useKakaoMapSdk.ts:131`
- script는 `document.head.appendChild(scriptElement)`로 삽입된다. 근거: `src/hooks/useKakaoMapSdk.ts:134`
- SDK load 이후 `window.kakao.maps.load(finishReady)`를 호출한다. 근거: `src/hooks/useKakaoMapSdk.ts:89`
- SDK timeout은 `10000ms`로 설정되어 있다. 근거: `src/hooks/useKakaoMapSdk.ts:12`, `src/hooks/useKakaoMapSdk.ts:74-82`

`autoload=false` 사용 자체는 긍정적이다. 다만 홈 첫 화면에서 지도 컴포넌트가 즉시 렌더링되므로, SDK 로드와 지도 초기화가 초기 화면의 주요 경로에 포함된다.

### 지도 초기화와 타일 로딩

`Map` 컴포넌트는 SDK가 ready가 되면 즉시 `new maps.Map(...)`으로 지도 인스턴스를 생성한다.

- 지도 생성: `src/components/Map.tsx:90-93`
- `tilesloaded` 이벤트에서 `tileStatus`와 `mapStatus`를 ready로 변경한다. 근거: `src/components/Map.tsx:100-106`
- `tilesloaded` 이벤트 리스너를 등록한다. 근거: `src/components/Map.tsx:119`
- 타일 timeout은 `8000ms`다. 근거: `src/components/Map.tsx:23`, `src/components/Map.tsx:123-126`

지도 타일은 외부 리소스이므로 앱 코드만으로 완전히 통제하기 어렵다. 다만 현재 UI에서는 전체 화면 지도 영역이 첫 화면의 중심이므로, SDK와 타일 응답 시간이 LCP 근처 지표에 영향을 줄 가능성이 높다.

### fallback 구조

지도 SDK loading 또는 map creating 상태에서는 지도 위에 큰 상태 패널을 보여준다.

- SDK loading/map creating 조건: `src/components/Map.tsx:162-165`
- loading 패널 렌더링: `src/components/Map.tsx:187-194`
- 지도 실패 시 error 패널 표시: `src/components/Map.tsx:198-224`
- 타일 timeout 시 안내 표시: `src/components/Map.tsx:228-232`

fallback store list는 지도 실패가 확정되고, store fetch가 성공했을 때만 나타난다. 근거: `src/pages/index.tsx:252`, `src/pages/index.tsx:272`

따라서 "지도보다 store list 또는 fallback store가 먼저 사용자에게 보이는 구조"는 아니다. 현재 구조는 지도 실패를 감지한 뒤에야 list fallback이 노출된다. LCP 관점에서는 지도 로딩 패널이 먼저 큰 영역으로 잡힐 가능성이 있다.

확인 필요: API 응답이 LCP element와 직접 연결되는지 여부는 Lighthouse의 LCP element와 네트워크 waterfall을 함께 봐야 한다.

## fallback / loading / empty / error 상태 개선 후보

현재 구조는 fallback 상태가 이전보다 정리되어 있지만, map loading과 store loading이 완전히 독립적인 사용자 흐름으로 보이기는 어렵다.

- 지도 loading은 `Map` 내부에서 처리된다. 근거: `src/components/Map.tsx:187-194`
- store loading/error/empty는 `StoreStatusPanel`에서 처리된다. 근거: `src/pages/index.tsx:63-141`
- store list fallback은 map 실패 이후에만 보인다. 근거: `src/pages/index.tsx:252`, `src/pages/index.tsx:272`
- marker 상태는 `Markers`에서 부모로 전달된다. 근거: `src/components/Markers.tsx:25`, `src/components/Markers.tsx:127-129`, `src/components/Markers.tsx:199-210`

개선 후보는 다음과 같다.

- 지도 SDK loading과 store loading 상태를 시각적으로 더 명확히 분리한다.
- 지도 타일이 느릴 때도 store list 일부를 먼저 보여줄지 검토한다.
- map 실패가 확정되기 전에도 최소 정보 fallback을 보여줄지 검토한다.
- production에서 debug 성격의 메시지 또는 개발 도구가 노출되는지 확인한다.

`ReactQueryDevtools`가 `_app.tsx`에서 항상 렌더링되어 있으므로 production bundle/노출 여부 확인이 필요하다. 근거: `src/pages/_app.tsx:5`, `src/pages/_app.tsx:30`

## 바로 수정 가능한 항목

다음은 코드 변경 범위가 비교적 작고, `v1.1.0`에서 바로 처리 후보로 볼 수 있는 항목이다. 이 문서에서는 수정하지 않았다.

## 결론

현재 LCP 약 `2.1s`가 유지되는 가장 강한 후보는 전체 화면 Kakao Map 중심 초기 렌더링 구조와 외부 지도 SDK/타일 의존성이다. 여기에 고용량 favicon, 일부 고용량 marker 이미지, 로컬 TTF 폰트, 초기 JS chunk 후보가 함께 모바일 성능과 체감 로딩에 영향을 줄 수 있다.

`v1.1.0`에서는 먼저 favicon, marker 이미지, 폰트, production devtools, map ready 이후 UI 지연 로딩처럼 변경 범위가 작은 항목을 검토하고, 지도와 목록의 구조적 재배치는 `v1.1.0`에서 충분히 측정한 뒤 `v2.0.0`으로 분리하는 편이 안전하다.
